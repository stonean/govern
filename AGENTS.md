# `govern`

The `govern` framework — a pipeline-driven spec-to-implementation flow scaffolded into adopted projects.

> **Agents:** this file is the committed home for project rules — append durable learnings to the matching section (Gotchas, Workflow, Boundaries, Code Style, Testing). Add a new section only when none fits.

## Constitution

See [constitution.md](constitution.md) — guiding principles, development pipeline, spec lifecycle, and quality standards that govern this project.

## Project Structure

The `framework/` directory is govern's source — everything that ships to adopted projects via `/govern`. It is laid out by IA primary purpose, not by file kind:

- `framework/constitution.md` — the constitution shipped to adopted projects (sync target of root `constitution.md`)
- `framework/rules/` — domain rule sets adopted projects can reference (security-backend, security-frontend, …)
- `framework/templates/spec/` — templates consumed by an agent during the pipeline (spec, plan, tasks, data-model, research, scenario)
- `framework/templates/project/` — project document templates consumed once at adoption (agents, claude-md, system, errors, events, project-readme, gitignore, inbox)
- `framework/commands/` — operational slash command sources only
- `framework/workflows/` — tech-stack-specific workflow files (lint, test, format) plus `registry.json` mapping stack selections to workflows
- `framework/bootstrap/` — the `govern.md` installer plus per-agent permission files at `bootstrap/configure/{key}.md`

When adding files under `framework/`, place them by purpose, not by extension.

## Tech Stack

`govern` is a text-first framework. The implementation is intentionally narrow:

- **Markdown** — every primary artifact (constitution, specs, plans, tasks, scenarios, rules, slash command sources, templates) is plain markdown; markdown is the source-of-truth *format* per [§text-first-artifacts](framework/constitution.md#text-first-artifacts). Format is not durability: `spec.md`, scenarios, and rules are durable sources of truth (with `plan.md` / `data-model.md` as design records), while `tasks.md` is ephemeral work-tracking — see [§tasks-phase](framework/constitution.md#tasks-phase).
- **Bash scripts** under `scripts/` — generators (`gen-*.sh`), the pre-commit hook installer (`install-hooks.sh`), and lints (`lint-*.sh`). All deterministic; no application logic.
- **GitHub Actions YAML** under `.github/workflows/` — CI configuration only.
- **Node.js / `npx`** — invoked at lint time for `markdownlint-cli2` only; not a build dependency, no `package.json` or `node_modules`.

There is no compiled language, no application runtime, no database, no service binary. An optional deterministic runtime is permitted under [§runtime-boundary](framework/constitution.md#runtime-boundary) but is not yet implemented (deferred to spec 022).

When tooling-language decisions arise (e.g., the runtime spec'd in 022, future binaries the framework might ship), prefer Rust. The `rmcp` crate is the reference MCP SDK, and the recent generation of Rust CLI tooling (`ripgrep`, `fd`, `bat`, `eza`, `helix`, `tokei`, `hyperfine`, etc.) has set the modern baseline for CLI UX patterns (single static binary, fast cold-start, sensible exit codes). Go is the credible alternative if development velocity matters more than production characteristics; other languages typically don't fit (distribution complexity, startup overhead, ecosystem fit).

## Workflow

- **Commit directly to `main` — this repo uses trunk-based development.** `govern` is live-on-main: the installer and everything `/govern` fetches track `main`, so there is no release branch and no feature-branch/PR step. When the user asks to land work, commit to `main` and push — do **not** branch first (the general "branch off the default branch before committing" default does not apply here). Reason: confirmed by the user 2026-06-11 after a `fix(003)` bootstrap fix was committed straight to `main`. How to apply: branch only when the user explicitly asks, or for a risky spike they want isolated; otherwise small commits reviewed in place on `main` are the norm, matching the existing history.
- **A `runtime/` change ships via a `gvrn-v<version>` tag — the commit alone reaches nobody.** The runtime is released (cross-compiled binaries + crates.io publish) by `.github/workflows/runtime-release.yml`, which triggers *only* on a pushed `gvrn-v*` tag. So a change under `runtime/` is not "landed" until you: (1) bump `runtime/Cargo.toml` `version` and add a matching `runtime/CHANGELOG.md` section (tag scheme `gvrn-v<MAJOR>.<MINOR>.<PATCH>`, a runtime bug fix is a patch bump); (2) commit to `main` and push; (3) `git tag gvrn-v<version>` at that commit and `git push` the tag. The tag pipeline runs the framework self-audit (`scripts/audit/run-all.sh`) as a **hard release gate** — any finding aborts the publish — so run it locally *before* tagging. The parity goldens store the version as a `{{runtime-version}}` placeholder (`runtime/tests/parity.rs` substitutes `env!("CARGO_PKG_VERSION")` at compare time), so a version bump needs **no golden re-bless**; do not `BLESS=1` for a version change (it would hardcode the literal version and destroy the placeholder). If parity fails on the version line after a bump, the cause is a stale `runtime/target/release/gvrn` — `cargo build --release` to refresh it, don't touch the goldens. Reason: surfaced 2026-06-30 — an `apply-manifest` exec-bit fix committed to `main` would have shipped to no adopter without the tag. How to apply: bump + CHANGELOG + commit + `gvrn-v<version>` tag + push tag, in that order; never bless goldens to absorb a version delta.
- Read `framework/commands/{name}.md` before recommending, describing, or disambiguating a slash command — don't guess from the name. Source files are authoritative; the generated `.claude/commands/gov/*.md` copies are produced by the pre-commit hook.
- `.govern.toml` is treated as a shared adopter-side database, not as a schema owned by any single spec. When a new spec adds a section/key (its own "table"), document it in that spec's body and behavior — do not generate a §cross-spec-impact signpost on spec 019 (or any prior `.govern.toml` spec) for the addition. Reason: surfaced 2026-05-10 while clarifying spec 020 — adding `[review] tech-stack-verified` did not warrant reopening 019. How to apply: only treat a `.govern.toml` change as cross-spec impact when it modifies an *existing* key already documented in another spec.
- **No dead references in live artifacts.** When renaming or removing a name (spec slug, capability, command, identifier, parenthetical descriptor, etc.), update every reference across **live artifacts**: `framework/`, `scripts/`, `runtime/` (including `tests/fixtures/`, `tests/golden/`, `tests/parity/`), `.github/`, `docs/`, `README.md`, `AGENTS.md`, and `specs/NNN-*/` (including done-spec bodies). Reason: a reader following a forward-pointer or back-reference in live artifacts must never land on an outdated name; spec bodies are living documents that represent current state. The sweep is mechanical: every change in the diff is the same find-and-replace token substitution applied uniformly across the artifact set above. Because the diff is uniform, it qualifies as a **mechanical sweep** under [§spec-lifecycle](framework/constitution.md#spec-lifecycle) and does NOT trigger the `done → in-progress` back-edge — done specs stay `done` after a rename sweep. How to apply: when renaming X to Y, grep the live-artifact paths above and update every hit in a single commit; do not bundle the sweep with unrelated edits (a non-uniform diff would qualify as a meaningful edit and reopen affected specs). Commit messages and published PR/release notes stay as written (git history is the audit trail of what was named when).
- **A behavior change needs a prose-claim sweep, not just an identifier sweep.** The "No dead references" rule above catches renamed/removed *names*; a change that alters what the system *does* additionally needs a sweep for stale *claims* about the old behavior — and those claims contain none of the changed tokens, so a path/identifier-scoped grep silently passes over them. Reason: surfaced 2026-06-18 — spec 031 split MCP registration per-agent (Auggie/Antigravity now surface a manual `mcp add` command instead of `/govern` writing a config file), and the spec's dead-reference sweep grepped only `.mcp.json` / `mcp_config.json` and passed, yet `README.md` still claimed `/govern` "registers it automatically" for every agent — false for two of three. How to apply: when a spec changes behavior (not just a name), enumerate the behavioral claims the change falsifies and grep for them **by meaning** — registration / "automatically" / "by hand" / "writes the file" phrasings, not just the changed path — across the live-artifact set, `README.md` especially since it narrates behavior to users. A prose-claim fix to `README.md` or `docs/` is docs-only and does not reopen the originating spec; a stale behavioral claim *inside a done spec body* is a meaningful edit and does reopen it (route it through `/{project}:amend` / the back-edge).
- **Cross-service reference edits are mechanical-class (non-reopening).** Adding, removing, or rewriting a cross-service reference — an inline body link whose target resolves to a registered `.govern.toml` `[services]` entry, plus the `references:` frontmatter that `scripts/gen-cross-service-refs.sh` harvests from it — does NOT trigger the `done → in-progress` back-edge under [§spec-lifecycle](framework/constitution.md#spec-lifecycle) (case (b) of the mechanical-edit rule): references are informative cross-service navigation, never dependencies. The `references:` index is generator-managed and **strictly distinct from `dependencies:`** (harvested by `scripts/gen-spec-deps.sh`); each generator owns its own frontmatter field and the two never share an index or enter the cycle check. Reason: spec 030. How to apply: a diff that only touches cross-service reference links and their harvested `references:` entries keeps a `done` spec at `done`, the same way a uniform rename sweep does — do not route it through `/{project}:amend`.
- **Run `/govern` per its spec — no ad-hoc prompts.** When executing `/govern` (here or in any adopter), do not insert confirmation prompts beyond those the spec specifies (project inputs, agent-selection prompts on `--add-agent` / first-run, the registry-driven migration prompts from `framework/migrations.toml`, the per-category workflow prompts). Specifically, do not stop to warn about uncommitted edits to update-strategy files, custom slash commands about to be removed by **Slash command cleanup**, or "data loss" from the `stale` → write-and-abort path. Reason: the spec already encodes safety — `.govern.toml` `[pinned] files` is the documented opt-out; the stale path writes upstream and aborts cleanly (recoverable from git); slash command cleanup is unconditional for unpinned files. Extra prompts duplicate information the spec already gives the user and stall routine runs. How to apply: trust the documented behavior. The canonical statement of this rule lives as a "Procedural fidelity" preamble at the top of §Instructions in `framework/bootstrap/govern.md` so it travels to every adopter via the self-update path; this AGENTS.md entry is the contributor-side mirror. If a real safety concern seems missing from the spec, raise it as a spec change (inbox entry or discussion) — not as a runtime prompt.
- **Use the `Write` tool, not Bash redirects, for `.govern.session.toml`.** Spec 023 added explicit `Edit(.govern.session.toml)` and `Write(.govern.session.toml)` permission entries so pipeline commands stop prompting on session-file updates. Those entries scope the **Edit** and **Write** tools only — they do not cover Bash redirects (`cat > … <<EOF`, `tee`, etc.), which fall under separate `Bash(...)` permissions. Reason: surfaced 2026-05-17 while running `/gov:target 025` — used a Bash heredoc and got a permission prompt despite spec 023's allowlist work. How to apply: when an agent writes the session file (or any `govern`-owned state file with a dedicated `Write(...)` allow entry), use the `Write` tool. Do not widen the Bash allowlist with patterns like `Bash(cat > * *)` to compensate — that grants write-anywhere-via-shell and defeats the per-path scoping. A narrow `Bash(cat > .govern.session.toml *)` would be safe, but reaching for the right tool is cheaper than maintaining two surfaces.
- **Use repo-relative paths in tool calls.** When invoking `Read`, `Write`, `Edit`, or `Bash` against files in this repo, use repo-relative paths (e.g., `.govern.session.toml`, `specs/027-bootstrap-migration-registry/spec.md`) — never `/Users/stonean/src/stonean/govern/...`. Reason: surfaced 2026-05-17 during `/gov:specify` — agent used an absolute path for the session-file write. Absolute paths leak the maintainer's home directory into transcripts, make diffs and error messages noisier, and break path-pattern matching for `permissions.allow` entries that are written relative. How to apply: every framework artifact (rule files, command sources, scripts, spec links) already uses relative paths; mirror that convention in tool calls. The only places absolutes are appropriate are temp files under `/tmp` or `/private/tmp` and explicit references to files in other working directories listed in the harness environment.
- **Never call specs "frozen archaeology" or use frozen-archaeology phrasing.** Specs are living documents that represent current state. Edits to a `done` spec body trigger the `done → in-progress` back-edge via `/{project}:amend` (or directly via `set-status` per the rule below); rename sweeps stay `done` because they qualify as mechanical edits under §spec-lifecycle. Reason: spec 023's `living-specs` scenario explicitly dropped the "frozen archaeology" exception from constitution §drift-prevention; surfaced 2026-05-24 when I wrote "outside frozen-archaeology specs" into spec 022's task 41 and the user corrected it. How to apply: when describing why a done spec shouldn't be edited, frame it in terms of the back-edge cost (it reopens the spec and goes through the pipeline again), not as "the spec is frozen." When describing a sweep that touches done specs, frame it as a §spec-lifecycle mechanical edit (uniform find-and-replace across live artifacts) that keeps the spec at `done`. Live framework files that still carry frozen-archaeology language (`framework/commands/analyze.md`, `framework/migrations/spec-and-plan-sunset.md`) are drift candidates from before the rule was dropped — flag them when adjacent, but do not bundle the sweep with unrelated work.
- **Re-open a `done` spec via `set-status` when the only intent is to reflect on-disk edits, not via `/{project}:amend`.** When the agent (acting on the user's behalf in conversation) has already added scenario files under `specs/{feature}/scenarios/` or appended lines to `spec.md` / `tasks.md`, and the spec's `status: done` is the only remaining inconsistency, invoke `set-status` (`from: done`, `to: in-progress`) directly and report the on-disk delta. Do not prompt `/{project}:amend` — the refinement loop expects an input to classify, and producing a synthetic re-open input is friction (the classifier will either insist on scenario shape and create a second file, or treat the input as a no-op). Reason: surfaced during the gvrn 0.10.0 session-consolidation cycle (see `specs/014-reclarify-backedge/scenarios/reopen-after-informal-edits.md` — Option A). How to apply: only when there is *no* new input from the user to capture. If the user is also adding new scenario content beyond what is already on disk, route the new content through `/{project}:amend` after flipping status — Option A is a shortcut for the re-open-only case, not a replacement for the scenario-add flow. The command-side safety net (Option B in `framework/commands/amend.md`) covers hosts without an agent in the loop.

## Gotchas

- Use `npx markdownlint-cli2` to run markdown linting — do not suggest installing it globally.
- The command generator substitutes `{project}` → `gov` and `{cli-config-dir}` → `.claude`, and writes the Claude-specific permission file (`framework/bootstrap/configure/claude.md`) as `configure.md` in the gov command directory.
- `framework/workflows/` files ship as-is — they are not generator inputs and have no `govern`-side `gov:workflows:*` counterpart. Adopting projects scaffold them via `/govern` (or `/gov:init`).
- `.claude/commands/gov/init.md` is the one exception to the generator rule — it is `govern`-specific (no source counterpart) and is hand-maintained. The generator leaves it untouched.
- Never `git add -A` / `git add .` in this repo — it sweeps untracked `/specify` drafts (e.g. an in-progress `specs/NNN-*/spec.md`) into the commit, which spec 017 (`tracked-specs-not-worktree`) forbids. Stage explicit paths.
- A new adopter-facing `scripts/gen-*.sh` generator wires into more than this repo's own `.githooks/pre-commit`. For adopters to actually receive and run it, two more sites are required: (1) a row in the **Shared Files** manifest in `framework/bootstrap/govern.md` (`update` strategy, `scripts/<name>.sh` → `scripts/<name>.sh`) so `/govern` copies it into adopter projects, and (2) an invocation in the adopter hook `framework/bootstrap/hooks/govern-pre-commit` (run it `--staged` so a commit only rewrites its own specs). Adding it only to this repo's `.githooks/pre-commit` exercises it on `govern`'s own commits — so CI and local dogfooding stay green while every adopter silently misses the generator until someone notices. Reason: surfaced 2026-06-17 — spec 030's `gen-cross-service-refs.sh` was wired into `.githooks/pre-commit` but not the manifest or adopter hook, so a `/govern` update in adopter projects never installed or ran it. How to apply: when adding a generator that derives adopter spec frontmatter, wire all three sites (this repo's hook, the manifest row, the adopter hook) in the same change; if it derives per-spec frontmatter independently it can take `--staged` directly, but if it needs a cross-spec view (like `gen-spec-deps.sh`'s cycle check) keep the full read and gate only the disk write on staged membership.
- `write_atomic_bytes` (the create-then-rename helper backing every primitive write) lands the destination at the fresh tempfile's mode — `0600` on Unix — discarding whatever mode the file should carry. For markdown/JSON this is invisible (git tracks only the exec bit), but a primitive that writes a file expected to be **executable** must re-apply the mode itself. `apply-manifest` mirrors the *source* file's `Permissions` onto the dest after each write (`mirror_source_mode`, `cp -p` semantics) because the **Shared Files** manifest ships `scripts/gen-*.sh` with `update` strategy and the `govern-pre-commit` hook must be able to exec them; the staging source carries `+x` because `extract-archive` preserves archive-header modes (`apply_unix_mode`). Reason: surfaced 2026-06-30 — a `/govern` run rewrote the generators to `0600`, failing the pre-commit hook (and a fresh adopter's `created` write would emit a non-executable generator + broken hook). How to apply: any new primitive writing an executable (or otherwise non-`0600`) file must restore the intended mode after `write_atomic_bytes` — do not assume the helper preserves it; mirror the source's `Permissions` when copying, like `apply-manifest` and `extract-archive` do.
- `runtime/rust-toolchain.toml` pins `components = ["clippy", "rustfmt"]`, and every `cargo` command run under `runtime/` makes rustup honor that file — auto-installing any missing component *mid-build*, a path that hits rustup's `detected conflict: 'bin/cargo-fmt'` bug and aborts the job. So any CI job that runs `cargo` under `runtime/` must pre-install those components via the `dtolnay/rust-toolchain` step's `components: clippy, rustfmt` input, so nothing is left for the runtime auto-install path to do. `runtime.yml` (which lints) always had it; the fix is to keep every other `runtime/`-touching workflow in sync. Reason: surfaced 2026-07-11 — the `gvrn-v0.16.0` release failed because `runtime-release.yml`'s `dtolnay` steps omitted `components:`; the `aarch64-unknown-linux-gnu` cross job's first `cargo` call (`cargo install cargo-zigbuild`) triggered the auto-install conflict. It was the first release tagged after the toolchain pin (commit 19d9229), so no earlier release exposed it. How to apply: when adding or editing a workflow that runs any `cargo` command from `runtime/`, give its `dtolnay/rust-toolchain@<ver>` step `components: clippy, rustfmt` matching `rust-toolchain.toml` — do not rely on rustup's toolchain-file auto-install to supply them.
- A new `gvrn` runtime primitive wires into more than `primitives/mod.rs` + `mcp/server.rs`. Building and exposing it over MCP needs `schema/primitives.rs` (Args/Result types), `primitives/<name>.rs` + `primitives/mod.rs` (`pub mod`), and `mcp/server.rs` (`TOOL_NAMES` + a `#[tool]` method). Three more sites are needed the moment a slash command *uses* it: `parser/mod.rs` (`PRIMITIVE_NAMES`) so a command's `## Instructions` can backtick the name without failing `lint-procedure-parseability`; `interpreter/mod.rs` (`dispatch_primitive` match arm) so `runtime exec` can call it (the path the parity tests drive); and `main.rs` (`Command` enum variant + dispatch) for the `gvrn <name>` CLI surface. Add the name to `framework/runtime-tools.txt` for the markdown-only-pipeline opt-in invariant and `lint-tool-coverage`. Reason: surfaced 2026-06-14 implementing spec 030's `resolve-references` — MCP exposure alone passes `cargo test` and `tests/mcp.rs` (which only checks `TOOL_NAMES ⊇ runtime-tools.txt`), so the gap stays invisible until a command references the primitive. How to apply: decide up front whether a command will reference the primitive; if yes, wire all six Rust sites + `runtime-tools.txt` in the same change.
- `<!-- audit:ignore-promotion -->` silences the framework self-audit's Family-9 (primitive-promotion) check for a prose-only `## Instructions` step, but it does **not** change how the runtime's exec parser reads that step. The parser (`parser/mod.rs`) marks any step naming a `PRIMITIVE_NAMES` token in a backticked code span as a primitive dispatch, and `lint-procedure-parseability` only rejects a step naming *two or more* primitives — a step naming exactly one is legal. So a host-responsibility step marked `audit:ignore-promotion` that mentions a primitive in backticks (e.g. a `--fix` action described "via `set-status`") gets dispatched by `runtime exec`, which then errors or performs an unintended mutation. Reason: surfaced 2026-07-12 — `/gov:analyze` step 13 (report render, a host responsibility) referenced `set-status` in backticks; the exec walker dispatched it and the `analyze` parity golden failed with a non-zero exit, even though the parseability lint passed and the audit ignored the step. How to apply: in a numbered `## Instructions` step, backtick a primitive name only when the step is meant to *dispatch* it. To reference another primitive without dispatching — a leverage note, a cross-reference — name it without backticks, or move the reference into non-step prose (the command's preamble or its Markdown-only reference), where multiple primitive names can be backticked freely. `audit:ignore-promotion` is about the audit, not the parser.

## Boundaries

- **No host-level enforcement of the deterministic (`gvrn`) path — it is a deliberate non-goal.** `/govern`'s State-A execution contract (spec 029, scenario `state-a-deterministic-path-forcing`) uses binding *procedure language* to get a wired runtime actually used. `govern` does **not** enforce it at the host level — no denying `curl`/`tar` while `gvrn` is registered, no State-A-only procedure variant with the shell stripped. Reason: host enforcement would block the legitimate non-primitive shell steps the bootstrap needs (gitignore `curl`, `git config`, `chmod`), is host-specific and a permanent per-host maintenance tax, and breaks [§runtime-boundary](framework/constitution.md#runtime-boundary)'s "neither path wraps the other" parity by demoting the markdown path. A model that ignores the binding contract falls back to the correct-but-pricier markdown path — an accepted outcome, not a gap. How to apply: do not propose or build host enforcement as a follow-up; if the procedure-level contract proves insufficient for a given model, the answer is a more capable model, not coercion.
- Never edit `.claude/commands/gov/*.md` directly — your changes will be overwritten the next time the generator runs. Edit the source under `framework/commands/` (or `framework/bootstrap/configure/claude.md` for the `configure` command). **When writing or rewriting that source, use placeholders, not substituted literals.** Write `{cli-config-dir}/` (not `.claude/`) and `/{project}:` (not `/gov:`); the generator substitutes them on the way to `.claude/commands/gov/*.md`. Reason: surfaced 2026-05-23 during the spec-022 `/gov:status` rewrite — I read the deployed (already-substituted) copy when drafting the rewrite and shipped hardcoded `/gov:` and `.claude/` literals to the source, which broke the `markdown-only-pipeline` CI on Family 4 of `scripts/audit/run-all.sh` (`placeholder-roundtrip`). How to apply: when in doubt, `git show HEAD:framework/commands/{name}.md` to see the source convention before editing; the `placeholder-roundtrip` audit is the post-hoc check, not the first line of defense.

## Design Principles

- **Never design framework features that depend on human diligence or discipline.** Any artifact section, frontmatter field, command behavior, or workflow step that requires an author to *remember* to fill it in, set a flag, update a doc alongside code, or otherwise be careful will fail in practice — silently and asymmetrically (the cases where it gets skipped are exactly the cases where it mattered most). When proposing a new input, ask "what happens when an author forgets?" If the answer is "the feature degrades silently," redesign the input as **derived** (extracted from existing artifacts, frontmatter, git history, code analysis) or don't ship it. Reason: surfaced 2026-05-06 when evaluating an optional `## Upgrade Impact` spec section as a way to capture cross-version migration notes; rejected on this principle and the topic was tabled to inbox until a derivable design is found. How to apply: this is a hard filter on framework proposals, not a tiebreaker — if the only viable design relies on author discipline, the right answer is to defer the feature, not to ship the disciplined version "for now."
