## Technical Decisions

### `/amend` becomes the status-mutation owner

The post-record step in `framework/commands/amend.md` is rewritten from a passive "Status warning" (informational only) into an active branch on the spec's current `status`:

- `draft` â†’ no mutation (existing behavior preserved).
- `clarified` / `planned` / `in-progress` â†’ mutate `status` to `draft` in the same write that appends the question.
- `done` â†’ refuse the entire recording (no question added) and redirect the user to `/{project}:amend`.

The mutation fires after the user accepts the refined question. The existing refinement loop is the consent point â€” adding a separate yes/no prompt at status-change time was rejected during clarify (see the spec's Resolved Questions) because it duplicates that consent.

Scenario-targeted `/amend` is unchanged. Scenarios have no status field, so the back-edge has nothing to mutate; the existing scenario-append path runs without touching the parent spec's frontmatter.

### Impact display before mutation

When `/amend` is about to mutate status (i.e., recording on a `clarified`/`planned`/`in-progress` spec), it surfaces what may need re-review afterwards. The display includes:

- The spec's prior status (already loaded for the gate decision).
- Existence and last-modified timestamp of `plan.md`, `tasks.md`, and `data-model.md` in the feature directory.
- The list of files in `specs/{feature}/scenarios/` (if the directory exists).
- A one-line note when the targeted spec appears in any other spec's frontmatter `dependencies` field â€” derived from a single scan of sibling specs' frontmatter (no body reads).

The downstream artifacts themselves are not modified; surfacing them is what tells the user that `/plan`'s new overwrite-protection (below) is the next safety net.

### `/amend` duplicate detection

The refinement loop already presents a final form for user approval. Before that approval, `/amend` compares the refined text against existing entries in `## Open Questions` using a normalized-whitespace match. On a hit, it reports the existing match and prompts skip-or-refine. Skip records nothing and exits without mutating status â€” the back-edge fires only when a question is actually recorded.

## Open Questions Resolved

All open questions were resolved during `/gov:clarify`; the spec's `## Resolved Questions` section records the rationale for each. The decisions that govern this plan:

- `/amend` owns the back-edge, mirroring `/elaborate`.
- The destination status is `draft` (the only state that tolerates open questions), not `clarified`.
- The trigger is the data (presence of an open question), not a user-supplied flag.
- No separate yes/no confirmation before status mutation â€” the refinement-loop acceptance is the consent.
- `## Resolved Questions` are never re-walked.
- `/plan` re-run safety is bundled into this spec because the back-edge cycle would otherwise destroy committed plan work on the next `/plan` run.

## Trade-offs

- **Symmetric back-edge ownership over central enforcement.** `/amend` mutates status rather than `/clarify` so the spec never sits in an internally inconsistent intermediate state. The cost is that `/amend` is no longer a pure record-only command. Mitigated by the existing acceptance-of-refinement consent point and the explicit impact display.
- **Recovery path retained even though normal usage cannot reach it.** Hand-edited frontmatter is rare but possible (manual fixes, migrations from other tools). Without the recovery path, `/clarify` would refuse those specs forever. Gated behind an explicit prompt so the system never silently fixes mismatched expectations.
- **`/plan` keep/replace prompt fires on every run, not only post-back-edge.** The detection logic does not know whether the user just came through a back-edge cycle. Always prompting is simpler and protects every re-run scenario, at the cost of one extra interaction on a fresh `clarified` spec where artifacts already exist. The "keep" default keeps that interaction lightweight.
- **No data-model file.** This spec changes command procedures and one frontmatter field that already exists. There are no new entities, schemas, or persisted shapes to document.
- **No new template changes.** The plan and tasks templates do not change shape; only the commands that consume them do.
