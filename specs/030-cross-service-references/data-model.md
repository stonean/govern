# 030 ‚Äî Cross-Service References Data Model

Data structures introduced by [030 ‚Äî Cross-Service References](spec.md). This file is the canonical source for the `[services]` registry schema (per the ¬ßdrift-prevention canonical-source table added in this feature).

## Service registry entry

Declared in `.govern.toml` under `[services.<alias>]`. The table is optional; an adopter with no cross-service references omits it entirely. Entries are added with the `/{project}:link` command, which validates the inputs and writes well-formed TOML; direct hand-editing remains possible.

| Field | Required | Type | Description |
| --- | --- | --- | --- |
| `<alias>` | yes | table key | Short, stable handle for the service (e.g., `api`). Author-chosen; need not match the repo name. |
| `repo` | yes | string | Canonical repository URL. The **identity** matched against a reference link's href to decide registration. |
| `path` | yes | string | Local checkout location, relative to the repo root or absolute. The location read for status resolution. |
| `description` | no | string | Human/agent-facing note on the service's purpose (e.g., "owns all data models; the system of record"). **Informational only** ‚Äî surfaced for orientation in `/{project}:status` and the registry; no `govern` behavior depends on it, so a blank description degrades to nothing. |

Example:

```toml
[services.api]
repo = "https://github.com/acme/api"
path = "../api"
description = "owns all data models; the system of record"

[services.frontend]
repo = "https://github.com/acme/frontend"
path = "../frontend"
description = "React management console"
```

Constraints:

- `repo` values should be unique across entries; a duplicate `repo` under two aliases is a registry-validation finding (the match becomes ambiguous).
- `path` is not required to exist on disk ‚Äî a missing path yields the `not-checked-out` outcome at resolution time, not a config error.

## Cross-service reference

Parsed from an inline body link whose href is a canonical spec URL. Not persisted in this raw form; it is the parse intermediate the harvester produces.

| Field | Type | Description |
| --- | --- | --- |
| `raw-url` | string | The link href as written in the body. |
| `repo` | string | Repository portion of the URL, matched against `[services].repo`. |
| `spec-slug` | string | The `NNN-slug` segment identifying the target spec. The stable identity. |
| `branch-ref` | string (ignored) | Any `/blob/<ref>/` segment. Recorded for navigation only; **never** part of identity or resolution. |

## References index (`references:` frontmatter)

A derived, generator-managed list on the consumer spec's frontmatter, parallel to `dependencies:` and strictly distinct from it. Body links are authoritative; the field is regenerated by `scripts/gen-cross-service-refs.sh`. Not hand-authored.

| Field | Required | Type | Description |
| --- | --- | --- | --- |
| `references` | no | list of entries | Empty or absent when the spec has no cross-service references. |
| `references[].service` | yes | string | The matched registry alias. |
| `references[].spec` | yes | string | The target `NNN-slug`. |

A reference whose repo matches no registry entry is still recorded (so `/{project}:status` can show it as `unregistered`); the `service` field is null/omitted in that case.

## Resolution outcome

Produced by the `resolve-references` primitive (and the markdown-only path) for each reference. The outcome is a closed enum decided by deterministic predicates; `status` is populated only on `ok`.

| Outcome | Meaning | `status` |
| --- | --- | --- |
| `ok` | Registered, reachable, target resolves, `status` readable and in-set | the linked lifecycle status |
| `unregistered` | href repo matches no `[services]` entry ‚Äî referencing without the registry | ‚Äî (not attempted) |
| `not-checked-out` | Registered, but `path` is missing or not a usable checkout | ‚Äî (`unknown`, informational) |
| `broken` | Registered + reachable, but the target spec does not exist or the URL is malformed | ‚Äî (`/{project}:analyze` Advisory finding) |
| `status-unreadable` | Target file exists but `status` is missing / malformed / out-of-set / a scenario | ‚Äî (`unknown`, surfaced) |

The allowed `status` values are the lifecycle set declared in constitution ¬ßtext-first-artifacts: `draft`, `clarified`, `planned`, `in-progress`, `done`.

### Resolution record

| Field | Type | Description |
| --- | --- | --- |
| `service` | string \| null | Matched alias, or null when `unregistered`. |
| `spec` | string | Target `NNN-slug`. |
| `outcome` | enum | One of the outcomes above. |
| `status` | string \| null | Linked lifecycle status; non-null only when `outcome` is `ok`. |
