# Code Location Index

## Note

The open questions below are downstream of [013-text-first-artifacts](../../013-text-first-artifacts/spec.md). Once 013 lands, the principle's "structured derived view" framing auto-resolves location and maintenance ("derived view, regenerated on demand"); the consumer question becomes a gate on building anything ("does any command actually need this?"). Granularity remains the one design choice that 013 does not auto-resolve. Defer this scenario's `/gov:clarify` pass until 013 is at `done`.

## Context

`/gov:implement` walks tasks, writes code, and verifies acceptance criteria, but does not record *where* the implementation landed. After a feature reaches `done`, there is no machine-readable mapping from acceptance criteria (or the feature itself) to the files, functions, or modules that satisfy them. This forces every downstream consumer — `/gov:validate` doing traceability, `/gov:status` answering "what surface area does this feature own?", `/gov:capture` and 011's brownfield triage answering "where does this bug live?" — to re-derive the mapping by reading code or asking the user.

A code-location index produced and maintained by `/gov:implement` would close that gap. The index becomes an authoritative artifact tying spec language to source-tree positions, surviving the implementation session that produced it.

## Behavior

- As `/gov:implement` works through tasks, it records the source files (and ideally functions/modules) it creates or modifies in service of each acceptance criterion.
- The index is written to a known location and is regenerated/refreshed on every `/gov:implement` run for the feature, so it stays current as implementation progresses.
- Downstream commands (`/gov:validate`, `/gov:status`, `/gov:capture`, brownfield triage) read the index rather than re-deriving the mapping.
- The index format is plain markdown — no new tooling, no JSON schema, no parsing dependency beyond what governance already requires.

## Edge Cases

- Code that is moved, renamed, or deleted after `done` — the index becomes stale unless something rewrites it. The maintenance question (regenerate vs. incremental) determines whether staleness is detected, accepted, or auto-healed.
- Acceptance criteria satisfied by configuration, infrastructure, or external artifacts (not source files) — the index needs a way to express "implemented by X" where X is not a code path.
- Brownfield features where `/gov:implement` was never run — the index must be backfillable by some other path (likely 011's triage flow) so brownfield specs are not permanently second-class.

## Open Questions

- **Location** — per-spec artifact (e.g., `specs/{NNN-feature}/code-locations.md`) or a single top-level `code-locations.md` index? Per-spec keeps each feature self-contained and survives spec moves; a top-level index makes cross-feature queries cheap. Trade-off depends on which consumer matters most.
- **Granularity** — per-feature, per-acceptance-criterion, or per-task? Per-criterion gives `/gov:validate` precise traceability but increases write volume; per-feature is cheap but answers fewer questions.
- **Maintenance** — regenerated by `/gov:implement` on every run, or appended incrementally? Regeneration is simple but loses history; incremental preserves history but drifts unless reconciled. How is staleness from renames, moves, or deletes detected and corrected?
- **Consumers** — confirm which commands actually read the index: `/gov:validate` for traceability and dead-criterion detection, `/gov:status` for surface-area summaries, `/gov:capture` for "where does this bug live," 011's brownfield triage for the inverse mapping (existing code → spec). Each adopted consumer drives a small contract addition to that command.
