- [ ] `/audit` command — framework-level self-audit for governance maintainers, distinct from `/validate` (which is feature-spec-scoped). Surfaced by the pre-v0.1.0 review passes (1–4) when ~30 of 45 findings turned out to be framework-level drift that `/validate` is not scoped to catch. The drift-prevention principles are now codified in `framework/constitution.md` §drift-prevention; `/audit` would automate the checks. Proposed scope: cross-doc claim consistency (README's spec-status table vs frontmatter, pipeline diagrams across docs, back-edge wording across constitution/introduction/project-readme); manifest parity (`/govern` vs `/{project}:init` scaffold the same files; `configure/claude.md` vs `configure/auggie.md` define the same canonical permission set in different formats); registry equivalence (every workflow file is in `registry.json` and vice versa; descriptions match frontmatter); placeholder roundtrip (no hardcoded `.claude/` etc. in command sources); template-validate alignment (every blocking validate check has a corresponding template scaffolding element); single-source-of-truth invariants (e.g., the open-question counting rule appears textually in only one location and other commands reference it). Run by maintainers before tagging releases. Drives `/specify` when picked up.

- [ ] Cross-version upgrade impact analysis — surfaced 2026-05-06 when a team member asked whether upgrading from a 3-revisions-old build to head would cause issues. The current pipeline tracks per-spec lifecycle but has no temporal aggregation: nothing answers "what shipped between point X and point Y, and what does an operator/integrator need to do about it." This is a real gap if `govern` is to displace Jira (versions/epics) and Confluence (release notes) combined. Three options considered: (A) **derived view from git tags** — `/gov:upgrade-report v1.0..v1.3` walks `git log` for spec status flips to `done` and scenarios added under reopened specs, no new artifact, but report quality is bounded by what specs already say; (B) **per-spec `## Upgrade Impact` section + `breaking: true` frontmatter + report command** — rejected per the no-human-discipline principle (see `AGENTS.md` §Design Principles), since an optional authoring section will go unfilled; (C) **Release as a fourth artifact tier** (`releases/{version}.md` alongside rules/specs/scenarios) carrying cross-spec migration narrative — heaviest, most Jira/Confluence-replacing, but introduces new lifecycle questions (when authored — at cut, or accreted as specs ship? who/what authors it?). Open: what does "revision" mean here — git tag, sprint, deploy? Artifact shape changes accordingly. Constraint for any future design: upgrade-relevant info must be extracted automatically (from acceptance criteria, frontmatter, env-var diffs, data-model changes, scenario additions) — no input that depends on an author remembering to fill in a section. Tabled 2026-05-06; revisit when `govern` is closer to general adoption and the question is forced by real upgrades.

- [ ] `append-task` MCP primitive has two bugs surfaced 2026-05-17 while creating `specs/023-govern-refinement/scenarios/living-specs.md` via `/gov:ask`. (1) **Default body uses `title` where it expects a slug.** When `body` is omitted, the primitive's spec says it emits `- [ ] Implement the behavior described in scenarios/{slug}.md` "derived from the title" — but the actual output substituted the full title into the `{slug}` slot. A title of `"Implement scenarios/living-specs.md"` produced the malformed line `- [ ] Implement the behavior described in scenarios/scenarios/living-specs.md.md`. The primitive needs an explicit `slug` argument, or it needs to extract the slug from the title via a defined rule (last path segment minus extension), not raw substitution. (2) **Numbering scheme is hardcoded to `## N.` top-level.** When a tasks.md uses a phased structure (e.g., 023's `## Phase A —` / `### N. Task`), the primitive finds no `## N.` matches and falls back to `## 1.` — collisional with the existing `### 1.` task and structurally wrong (h2 vs the file's h3 task convention). Real fix: detect existing structure (phased vs flat), match the convention, or accept a `parent-heading` arg. Both bugs required manual cleanup post-invoke; the primitive's "atomic-write" guarantee meant the broken state landed on disk before the operator could inspect.

- [ ] `read-tasks` and `check-stuck` MCP primitives have structural-assumption bugs in the same family as the `append-task` issue above. Surfaced 2026-05-17 during `/gov:implement` against 023's living-specs scenario. (1) **`read-tasks` returns an empty task list when `tasks.md` uses a phased structure.** 023's tasks.md uses `## Phase A —` / `## Phase B —` / `## Phase C —` with `### N. Task` headers inside each phase. The primitive's parser only matches `## N.` top-level numbered headers and returns `tasks: []`, even though 18 tasks exist. The implement walker that relies on `read-tasks` to identify the next incomplete subtask is blind on these specs. (2) **`check-stuck` measures from the wrong baseline.** It computes `since-sha` from the *first* time the spec entered `in-progress` rather than the most recent reopen, so a spec that went `in-progress → done → in-progress` (via `/gov:ask`'s back-edge) inherits every commit from the original implementation window. 023's first task-18-related commit attempt fired `stuck: true` with `commit-count: 8` because the original tasks-1-through-17 commits still counted, even though task 18 is brand-new and hasn't been touched. Real fix: identify the most recent `in-progress` transition (git log on the spec.md frontmatter line, or a dedicated marker) and count from there. All three primitive bugs (`append-task`, `read-tasks`, `check-stuck`) share the same root cause: hardcoded structural assumptions about tasks.md and the in-progress history that don't survive contact with real specs.
