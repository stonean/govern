# Inbox

- **`check-stuck` over-reports false positives:** The primitive at `runtime/src/primitives/check_stuck.rs` sets `stuck = count >= threshold` based purely on the number of commits touching `tasks.md` since the most-recent `in-progress` transition. The implement.md contract specifies a second condition that is not implemented: `stuck` should only fire when the same task is still the first incomplete one (no checkbox flipped to `- [x]` between those commits for that task). Effect: once 3+ commits land on `tasks.md` — even when each flips a different box — `stuck: true` fires on every subsequent `/anvil:implement` run for the remainder of the feature, training agents and users to dismiss the warning. Fix: in `count_commits_touching` (or alongside it), track the index of the first `- [ ]` group across the walked commit range and only mark stuck when that index has not advanced. The existing `counts_commits_since_in_progress` test would need a sibling that asserts `stuck: false` when checkboxes have been flipped across the threshold-count commits. (Origin: anvil/017-pagination implement session 2026-05-17 — second occurrence reported by user)
